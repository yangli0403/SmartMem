# 产品规格说明书：SmartMem 高级记忆系统

## 1. 产品愿景

**目标**：为 AI 智能体（尤其是 AI 陪伴和车载助手）提供一个先进、高效且可扩展的长期记忆系统。SmartMem 旨在取代现有简单的记忆模块，通过引入多策略检索、时序理解、动态遗忘和 LLM 驱动的知识巩固，使智能体能够建立更深层次、更个性化、更具智慧的用户理解。

## 2. 核心功能列表 (MVP)

| 优先级 | 功能模块 | 核心描述 |
| :--- | :--- | :--- |
| **P0** | **可切换的混合检索** | 实现一个基于环境变量的开关，允许在 `simple` (SQL LIKE) 和 `hybrid` (BM25 + 向量) 模式间切换。MVP 阶段 `hybrid` 模式至少包含 BM25 检索。 |
| **P0** | **时序有效性窗口** | 数据库 Schema 增加 `validFrom` 和 `validUntil` 字段，检索时自动过滤无效记忆，并能从对话中提取时间信息。 |
| **P0** | **动态遗忘机制** | 实现基于艾宾浩斯遗忘曲线的后台任务，根据访问频率和重要性动态调整记忆的“新鲜度”，代替硬编码的删除规则。 |
| **P1**| P1 | LLM 驱动的记忆巩固 | 创建后台服务，支持**定期扫描**或通过**API手动触发**，对零散的情景记忆进行巩固，利用 LLM 提炼并生成高阶的语义记忆（核心事实）和行为模式。 |
| **P1** | **分层上下文管理** | 建立“核心记忆”层，将 LLM 巩固生成的用户画像摘要常驻于 Prompt，确保个性化一致性。 |
| **P2** | **Reflect 综合推理** | 在 `hybrid` 检索模式中增加一个 LLM 调用，对多路召回的结果进行二次分析和总结，输出最相关的核心信息。 |

## 3. 用户故事

- **作为用户**，当我对 Agent 说“我下周要去趟北京”，我期望 Agent 在下周我再次对话时能记得我在北京，并能主动提供当地天气或新闻，而不是遗忘这个有时效性的信息。
- **作为用户**，当我反复在早上8点让 Agent “导航去公司”后，我期望 Agent 能学习到我的通勤习惯，在我下次早上8点打开 App 时能主动询问“是要去公司吗？”。
- **作为用户**，当我告诉 Agent “我喜欢喝拿铁”后，我期望它在后续推荐咖啡时能记住这个偏好，而不是每次都重新问我。

## 4. 技术栈总结

| 类别 | 技术选型 | 理由 |
| :--- | :--- | :--- |
| **语言/框架** | TypeScript / Node.js | 与 SmartAgent3 现有技术栈保持一致，便于集成。 |
| **数据库 ORM** | Drizzle ORM | 继承自 SmartAgent3，提供类型安全的 SQL 查询构建。 |
| **数据库** | MySQL / TiDB | 兼容 SmartAgent3 的数据库选型。 |
| **关键词检索** | `mini-search` | 轻量级的纯 JS 库，易于集成，无需额外服务，适合实现 BM25 算法。 |
| **向量嵌入** | `transformers.js` (Xenova) | 可以在服务端直接运行，无需依赖外部 Python 服务或 API，适合 MVP 快速验证。 |
| **向量存储/检索** | 数据库原生方案 (如 `pgvector`) 或内存暴力计算 | 长期方案推荐 `pgvector`。MVP 阶段，在数据量不大时，可先在内存中进行暴力余弦相似度计算。 |
| **核心依赖** | `@langchain/core`, `@langchain/openai` | 继承自 SmartAgent3，用于与 LLM API 交互。 |

## 5. 关键假设与风险

- **假设 1**：混合检索带来的延迟（尤其是 Reflect 步骤）可以通过产品设计（如流式输出、垫话）来缓解，用户可以接受为了更高质量回复而增加的等待时间。
- **假设 2**：LLM 在记忆提取和巩固任务中的表现是可靠的，能够准确地从对话中解析出结构化信息和高阶模式。
- **风险 1**：**成本增加**。混合检索和记忆巩固会显著增加 LLM API 的调用量，需要对成本进行监控和优化。
- **风险 2**：**复杂性提升**。新的记忆系统引入了多个后台任务和复杂的检索逻辑，对系统的稳定性和可维护性提出了更高要求。
- **风险 3**：**数据回填**。为现有记忆生成向量嵌入是一个需要仔细规划和执行的一次性任务。
